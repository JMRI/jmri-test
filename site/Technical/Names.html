JMRI Names and Naming

This page discusses how JMRI objects are named, how those names are used to reference the objects (hardware and software), and how user-readable names are used.

What's in a name?

Why do we need names at all, rather than pointers? There are several uses that seem important:
  1. When working with user input, e.g. typing a number in a field, the code will need to find how to map the user-provided info onto objects. Sometimes the user will want to name objects with arbitary user names, e.g. "East Lockport Turnout". These can't be known until their mapping to hardware has been made. But other times these will be de-novo indentifications that the code must understand, e.g. "LocoNet Turnout 23".
  2. We imagine that configurations will be stored in XML files, in a symbolic form. Names are a convenient way to connect objects and object references in that kind of configuration.

Items with names

In general, all hardware-related objects get names:
  • Aspects for signals
  • Blocks
  • Inputs (A general type of input, as opposed to a block-occupancy detector)
  • Locomotive Decoders (which includes "virtual" addresses like consists)
  • Outputs
  • Power districts and subdistricts
  • Signal heads
  • Turnouts

Unique names also carry an indication of which hardware system they are bound to:

  • C/MRI
  • Direct drive of the rail signal from the computer
  • Internal objects with no hardware equivalent
  • LocoNet (Digitrax)
  • NCE System One and similar
  • XpressNet (Lenz)

System names

A system-name is formed from a single letter representing the hardware system, followed by a single letter indicating the type, followed by system- and type-specific string. The string is meant to be related to the hardware addressing, but is otherwise unconstrained.

Examples:

  • LT23 - LocoNet Turnout 23.
  • CB134 - C/MRI input block 134
Note that there is no assumption of pattern to the names; they don't have to be assigned monotonically, nor are there restricted to a single system.
